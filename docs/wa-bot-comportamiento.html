4 · Los 3 modos (interruptor /api/bot/mode)

ModoToolsmax_tokensComportamientoCuándo
normal 7600 Bot completo: busca, cotiza, registra pagos, manda fotos. Operación sana (GREEN)
conservative 5350 Responde pero no crea cotizaciones ni registra pagos; respuestas más cortas, fotos solo si las piden. Dudas / volumen (YELLOW)
escalate_all 0— Cero IA. Marca todo como escalado para que conteste un humano. Riesgo / incidencia (RED)

Persistido en tabla app_settings clave bot_reply_mode. Lectura en cada mensaje (get_bot_mode, main.py:3325). El modo recorta la lista de tools y los tokens dentro de ai_reply (main.py:1735).

5 · Dónde se edita cada cosa (capas del comportamiento)

CapaQué gobiernaDónde viveCómo se cambia
SYSTEM_PROMPT Tono, prohibiciones A-N, anti-baneo Meta, cribar por uso, brevedad, cero emojis main.py:66-203 rsync + docker compose up -d --build
bot_knowledge Datos de cara al cliente: números WA, horarios, ubicación, envíos, pagos tabla bot_knowledge en conversations.db (volumen) UPDATE SQL en vivo → inmediato, sin rebuild
Tools + flujo Qué puede hacer y las guardias (escalado, horario, estados) main.py (lógica Python) rsync + rebuild

Gotcha: el seed de bot_knowledge en el código solo se inserta si la tabla está vacía. En prod ya tiene filas, así que editar el seed NO cambia nada — hay que hacer UPDATE sobre la DB viva (ver CLAUDE.md → "wa-bot: la verdad de cara al cliente vive en SQLite").

Para perfeccionar comportamiento/tono → editar SYSTEM_PROMPT (rebuild). Para datos al cliente → UPDATE bot_knowledge (en vivo). Para nuevas capacidades → añadir tool en CLAUDE_TOOLS + su función backend.
